# Handoff: CTO Audit Workflow Implementation

## What You're Building

A Claude Code-powered audit system where CTOs can interactively work through the Ultimate CTO Checklist with Claude's help. CTOs create a private workspace repo, add this checklist as a git submodule, configure their org/projects, and audit items one by one with automated checks and evidence capture.

## Required Reading

Read these files first, in this order:

1. `docs/plans/2026-02-04-cto-audit-workflow-design.md` — Approved design document. Contains architecture, workspace structure, config formats, scope model, skill list, dashboard integration, and all design decisions.

2. `docs/plans/2026-02-04-cto-audit-workflow-implementation.md` — Detailed implementation plan. 25+ tasks across 6 phases with exact file paths, code, and verification steps.

3. `CLAUDE.md` — Project conventions and structure.

## Critical Context

### Items.yaml has three formats in the wild

**Old (sections 01–12):** `section` is an object with `id`, `name`, `description`
```yaml
section:
  id: "01"
  name: Git Repo Setup
  description: ...
```

**Newer (sections 33–37):** `section` is a string, top-level `title` and `description`
```yaml
section: 33-feature-flags
title: Feature Flags & Rollouts
description: ...
```

**Newest (section 38):** Same as newer, plus `check_type`, `auto_checks`, `ask_user`, `pass_criteria`, `fail_criteria` on items.

When adding `default_scope`, check each file's format first:
- Object-style: add inside the section object
- String-style: add as top-level field after `description`

### Dashboard already handles both formats

`dashboard/lib/checklist.ts` normalizes both. New fields must be optional to maintain backward compatibility. The dashboard uses Next.js 16, React 19, TypeScript, Tailwind, and the `yaml` npm package.

### Skills are directories with SKILL.md

Claude Code skills live in `skills/<skill-name>/SKILL.md`. Directory name becomes the slash command (e.g., `skills/audit-init/` → `/audit-init`). Kebab-case only, no spaces.

### Scope defaults by section

Defined in the implementation plan Task 1.1. Key pattern:
- Sections 01–14: mostly `project`
- Sections 15–16, 20, 25, 35, 37–40: `org`
- Sections 11, 13, 24, 26, 29: `both`

### Workspace structure (what the CTO ends up with)

```
my-company-audits/
├── CLAUDE.md                 # Master index (generated by /audit-init)
├── STATUS.md                 # Dynamic stats
├── org.yaml                  # Org config
├── docs/                     # Instruction files for Claude
├── projects/*.yaml           # One per project
├── audits/
│   ├── _org/YYYY-MM-DD/     # Org-level results
│   └── <project>/YYYY-MM-DD/ # Project results (one .md per item)
├── waivers/                  # Documented exceptions
├── custom-items/             # Company-specific checklist items
└── checklist/                # Git submodule → this repo
```

### Audit result files

Markdown with YAML frontmatter, one file per audited item:
```markdown
---
item_id: GIT-005
title: Branch protections configured
status: pass
severity: critical
section: 01-git-repo-setup
audited_at: 2026-02-04T10:30:00Z
auditor: claude-session
---

## Evidence
[command outputs]

## Notes
[CTO's comments]
```

Status values: `pass`, `fail`, `partial`, `skip`, `not-applicable`, `blocked`

## What Not To Do

- Don't change existing items.yaml content beyond adding the `default_scope` field
- Don't break the dashboard's current browse-only functionality
- Don't mix skill logic — each skill should be self-contained in its SKILL.md
- Don't create the CTO workspace inside this repo — it's always a separate repo
- Dashboard audit pages should degrade gracefully when `AUDIT_WORKSPACE` is not set

## Verification

After each phase:
- Phase 1: `grep -r "default_scope" checklist/*/items.yaml | wc -l` should be 41. Dashboard `pnpm build` should succeed.
- Phase 2–4: Each SKILL.md should be a valid skill file with `---` frontmatter containing `name` and `description`.
- Phase 5: `AUDIT_WORKSPACE=/path pnpm dev` should show audit pages. Without the env var, existing pages should work unchanged.
